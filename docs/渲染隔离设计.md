# CTF|认证 渲染隔离设计

## 第一性结构

游戏进程只作为被观察目标；驱动、采集守护进程和渲染进程均不注入游戏，不加载游戏地址空间内的 SO，不修改 GOT/PLT、VTable、UE ProcessEvent 或渲染交换链。

```text
delta_kernel.ko -> root native reader -> lock-free snapshot -> ImGui renderer
```

## 降低游戏层可见面的原则

- 不使用 ptrace。
- 不访问 `/proc/<pid>/mem`。
- 不向游戏进程注入共享库。
- 不 Hook EGL、Vulkan、OpenGL ES 或 UE 函数。
- 不在游戏线程执行代码。
- 渲染器与读取器分进程，渲染器只读取固定快照。
- 驱动节点权限为 `0600`，普通游戏 UID 无权打开。
- 读取以 4 KiB 为硬上限，按帧合并请求，避免高频 syscall 噪声。

## Android 渲染后端选择

首选独立 SurfaceFlinger/SurfaceControl 图层加 EGL/OpenGL ES 3，不申请游戏进程内上下文。普通应用 UID 无法直接枚举其他 SurfaceFlinger 客户端，但具有系统调试权限的检查仍可能观察到独立图层，因此不存在“任何检测模型下绝对不可见”的技术保证。

备用后端为 Android Overlay Service；它实现简单，但会留下包名、AppOps 和 `TYPE_APPLICATION_OVERLAY` 窗口记录，不作为主路线。

当前 `imgui_core` 已与采集层完全解耦，下一阶段只需实现 SurfaceControl/EGL 平台后端，不改读取协议和业务面板。
